# Forms in the WKS Platform

Forms in the WKS Platform are essential user interface components designed to collect and input structured data. They enable users to interact with processes efficiently, support data-driven decisions, and streamline workflow execution across the platform.

## Key Features of WKS Forms

### 1. Data Input
Forms support the entry of various data types including:
- Text
- Numbers
- Dates
- Boolean values
This functionality ensures that all relevant task or process data is captured accurately.

### 2. Customisation
Forms are highly customizable:
- Define input field types (e.g., dropdowns, checkboxes, radio buttons)
- Adjust layout and styling to match process needs
- Apply validation rules to maintain data integrity

### 3. Dynamic Fields
Dynamic behaviour allows fields to:
- Show or hide based on user inputs
- Adapt according to predefined conditions
This provides a responsive and contextual user experience.

### 4. Integration with Platform Components
Forms seamlessly integrate with other WKS features, including:
- Workflow processes
- Analytics dashboards
- Data management tools
This ensures a connected and consistent experience across the platform.

### 5. Data Validation
Validation logic can be embedded to:
- Enforce field requirements
- Check for correct formats and input ranges
- Prevent submission of incomplete or incorrect data

### 6. Access Control
Role-based access control (RBAC) ensures:
- Only authorised users can view or edit specific forms
- Sensitive data remains secure and compliant with policies

### 7. Workflow Integration
Forms function as interactive steps within workflows, allowing:
- Task-specific data capture
- Guided user interactions through predefined process stages

### 8. Structured Data Collection
Captured data can be used for:
- Reports and dashboards
- Process optimisation
- Audit trails and compliance tracking

---

Forms are a foundational component of the WKS Platform, enabling efficient, secure, and intelligent interaction with automated workflows.

## Form document (the Standard)

A form is a configuration document validated against [`form.schema.json`](https://github.com/wkspower/wks-platform/blob/develop/packages/case-config-schema/schemas/form.schema.json).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `key` | string | ✅ | Unique form key, referenced by `CaseDefinition.formKey`. |
| `title` | string \| null | | Display title. |
| `toolTip` | string \| null | | Optional tooltip. |
| `structure` | object | ✅ | The [Form.io](https://form.io) schema tree. |
| `schemaVersion` | string | | Standard version (see [Configuration Versioning](../04-Case%20Definition/06-configuration-versioning.mdx)). |

### `structure`

`structure` is a Form.io schema:

- `display` — `"form"` or `"wizard"`.
- `components` — the component tree (text fields, selects, layout containers, etc.).

The component tree is validated **permissively** — Form.io is an open component model and real trees include typeless container/layout components. On top of the standard component set, WKS adds one non-standard component, `recordtype`, documented in [the recordtype extension](./02-recordtype-extension.mdx).
