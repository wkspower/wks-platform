# Step by step for local development setup 🛠️

## 📋 Prerequisites

- Docker and Docker Compose (v2) installed.
- That's it for the default path — images are pulled from the public GHCR registry.
- *Only if you intend to build from source:* Java 21 + Maven, and Node.js + Yarn. The
  Docker builds are multi-stage, so these are **not** required just to run the platform.

## Step 1: Clone and configure 📥

```bash
git clone https://github.com/wkspower/wks-platform.git
cd wks-platform
cp .env-sample .env
```

The defaults in `.env` target `localhost` and work out of the box.

## Step 2: Start the platform 🚀

### Minimum (recommended) — pulls pre-built images, no build

```bash
docker compose up -d
```

This starts the core backend plus the case portal (the `portal` profile is enabled by
default in `.env`). First start pulls images and may take a few minutes to become healthy.

### With seeded demo data

```bash
docker compose --profile portal --profile demo up -d
```

### With notifications (Kafka + email/websocket/Novu)

```bash
KAFKA_ENABLED=true docker compose --profile portal --profile notifications up -d
```

### Build from source instead of pulling

```bash
docker compose up -d --build
```

The multi-stage Dockerfiles compile the Java/Node apps inside Docker — **no host
`mvn`/`yarn` step needed**.

## Step 3 (optional): Custom domain

Skip this if you only use `localhost`. Otherwise copy/adjust values in `.env`
(`KEYCLOAK_TOKEN_URL`, `REACT_APP_*`, `MINIO_HOST_EXTERNAL`, `WEBSOCKET_URL`, the Novu
keys, and the default usernames/passwords) to match your domain, then bring the stack up.

## Step 4: Service configuration

If you changed the domain, update the `wks-portal` Keycloak client's **Web origins** and
**Valid redirect URIs** to match. For a plain `localhost` setup the defaults already work.
