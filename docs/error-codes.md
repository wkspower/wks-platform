# WKS Error Code Registry

Canonical reference for the `WKS-XXX-NNN` codes WKS Power emits over the wire. These strings are a **wire contract**: SI integrators, frontend snapshot tests, and customer runbooks grep against the exact text. Once a code ships it must never be reused for a different meaning, never renumbered, and never silently retired.

Last synced: v2-develop @ `cbec4129b` (after merges of 3-4, 3-6, 4-3).

## Source of truth

The canonical enum lives at:

- `backend/src/main/java/com/wkspower/platform/domain/exception/ErrorCode.java`

HTTP status mapping is intentionally NOT on the enum — it lives in `backend/src/main/java/com/wkspower/platform/api/GlobalExceptionHandler.java` so the domain layer stays transport-free.

This document is generated by hand for now (see `docs/error-codes-automation-spec.md` for the planned generator). When you add or change a code:

1. Edit `ErrorCode.java` (Javadoc + constant).
2. Wire it into the appropriate thrower / validator.
3. Update this doc until the generator lands.

The flat enum shape (no sub-enums) is deliberate — it keeps "find every code" a one-line search.

## Categories

Prefixes are **namespace, not lifecycle phase.** A single prefix may host both deploy-time and runtime codes (e.g. `WKS-MAP` carries deploy-time validation 001–007 and runtime miss 404). This is deliberate: renumbering codes is the most expensive thing you can do to a wire contract, so the band's purpose is to group by domain owner, not by when in the request lifecycle the code fires. Decided 2026-05-06.

The enum currently uses seven prefixes:

| Prefix | Band | Domain |
| --- | --- | --- |
| `WKS-API` | 001–005, 401, 403, 404, 413, plus security/bootstrap 050–055 (literals) | Transport / request-shape / auth + production-bootstrap fail-closed |
| `WKS-CFG` | 000–099 | CaseType + BPMN deploy-time validation aggregate |
| `WKS-MAP` | 001–009, 404, 405 | Mapping Layer validation (Story 4.2 + 4.3.1) + runtime miss (Story 4.3 + 4.3.1) |
| `WKS-STG` | 001–011, 099 | Stage lifecycle runtime + stage-scoped status sets |
| `WKS-VER` | 001–099 | CaseType Version Registry (Story 3.4 / Decision 20) |
| `WKS-RTM` | 409, 500 | Runtime conflict / last-resort |

## WKS-VER — CaseType Version Registry (Story 3.4 / Decision 20)

Band: `WKS-VER-001..099` reserved for version-registry-related errors. Future stories (3.5 bootstrap, 3.8 blast-radius validator, 3.9 rebase tooling) draw from this band. Per `feedback_error_codes_are_wire_contract.md`, never reuse a code in this band for a different meaning.

| Code | HTTP | Meaning | Thrower(s) |
| --- | --- | --- | --- |
| `WKS-VER-001` | 503 (Retry-After: 5) | `CaseService.create` called for a CaseType that is registry-visible (in-memory) but has no row in `case_type_versions` yet — partial-failure recovery state, registry-not-yet-primed, or startup race. The 503 + Retry-After signal is "transient, retry safe" — the most actionable contract for SI operators (Story 3.4.1 AC4 flipped this from 409, which misleads as "client conflict"). Story 3.5's bootstrap migration closes the gap for pre-3.4 CaseTypes by materialising v1 rows. | `domain/service/CaseService.java`, `api/GlobalExceptionHandler.java` |

## Known gaps and discrepancies

Surfaced for follow-up. Items 1, 2, 5, 6 have resolutions deferred to the **Registry hygiene pass** prerequisite story in `wks-platform2/docs/error-codes-automation-spec.md` (AC1–AC4). That story is sequenced after stories 3-4, 3-6, and 4-3 merge (now satisfied as of the sync line above).

1. **`WKS-API-050..054` are not in the enum.** Emitted as raw string literals from `security/AdminUserSeeder.java`, `security/JwtTokenProvider.java`, and `security/SecurityConfig.java`. They never reach the `ApiResponse.error.code` envelope (startup logs / `IllegalStateException` messages), but they are still wire-grep-stable strings customers will see in support tickets. → **Resolved by spec AC1**: promote to enum, refactor security wiring.

2. **`WKS-CFG-101..103` exist only in test fixtures.** Defined only in `backend/src/test/java/com/wkspower/platform/api/error/ConfigProbeController.java` for the `ConfigProbeWebMvcTest` envelope smoke test. → **Resolved by spec AC2**: renumber to `WKS-CFG-901..903`; carve `WKS-xxx-900..999` as the documented test-only sub-band across all prefixes.

3. **`WKS-CFG-014..019` are reserved gaps.** Documented in the enum Javadoc as "reserved for Story 2.2 (BPMN validation) — leave gaps, do not renumber." Listed here so the registry generator does not later misread them as missing.

4. **`WKS-CFG-023..026, 030, 034..098` are unallocated.** Available for future stories. The 020-band is BPMN, 030-band is stages — preserve clustering when allocating.

5. **`WKS-MAP-002` emit site.** The enum Javadoc names this code, but a direct grep over `MappingValidator` did not show a literal `WKS_MAP_002` reference. → **Resolved by spec AC3**: mark RESERVED with `// TODO: emit site` comment; reclaim slot at Phase-0 if still unused.

6. **`WKS-CFG-029` is reserved by Story 4.2 but emitted by Story 3.8.** Cross-story references like this are easy to lose. → **Resolved by spec AC4**: adopt `// RESERVED-FOR-STORY-X.Y` annotation convention; sweep enum for other implicit reservations; generator renders reserved codes in a separate section.

7. **`WKS-STG-007` is unallocated; `WKS-STG-012..099` are unallocated.** Preserve clustering when allocating: 001–004 are lifecycle, 005–011 are stage-scoped status sets (Story 3.6), 009 reserved for 3.7/3.8.

8. **`WKS-STG-009` reserved cross-story (3.7 / 3.8).** Same pattern as `WKS-CFG-029` (item 6) — slot allocated by Story 3.6 but not emitted by it; consumers are the future live-append (3.7) and mutate-class enforcement (3.8) paths. → **Folds into spec AC4**: same `// RESERVED-FOR-STORY-X.Y` annotation sweep.

9. **`WKS-MAP-404` is the first runtime miss code in the enum.** Story 4.3 introduced an HTTP-status-suffixed code (`404`) inside a band that was previously deploy-time-only (`WKS-MAP-001..007`). → **Resolved 2026-05-06**: prefixes are namespace, not phase. The mixed band is accepted; see Categories section. The generator does not need to surface a runtime/deploy-time axis — the per-code Meaning column carries that information.

10. **`WKS-VER-` is a new prefix introduced by Story 3.4.** The band `WKS-VER-001..099` is reserved for version-registry errors per the enum Javadoc. Listed here so the registry generator picks up the new prefix in its allow-list. → **Resolved implicitly by spec AC1–AC4**: a generator that reads `ErrorCode.java` discovers prefixes automatically; documenting here as a reminder for the manual generator path.

11. **`BackendSignalRouter` emits `"WKS-MAP-404"` as a string literal in three sites** (`addKeyValue("wksErrorCode", "WKS-MAP-404")` × 2, `publish(..., "WKS-MAP-404", ...)` × 1) instead of routing through `ErrorCode.WKS_MAP_404.wire()`. → **Resolved by spec AC1 (Cluster B)**: there is no log-only exception. All wire-grep-stable strings must be enum-routed; refactor the three audit/log sites alongside the security-band cleanup. Generator will fail CI on raw `"WKS-..."` literals outside test fixtures.

12. **`GlobalExceptionHandler` HTTP-status switch references `"WKS-STG-002"` / `"WKS-STG-004"` as string literals.** Pre-existing pattern (Story 3.1) where the handler's status mapping uses raw wire strings. → **Resolved by spec AC1 (Cluster C)**: same principle as Clusters A and B — refactor to `ErrorCode.WKS_STG_002.wire().equals(...)` (or whichever comparison shape fits), since AC1's CI guard would otherwise fail on landing. Folded into the hygiene pass so the principle ships with no pre-existing exemption.
